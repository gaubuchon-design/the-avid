# ADR-006: UX Safety Affordances for Agentic Editing

**Status:** Accepted
**Date:** 2026-03-08
**Authors:** UI Components Team
**Supersedes:** None
**Related:** ADR-005 (Orchestration Safety)

## Context

ADR-005 established the server-side Plan-Preview-Approve-Execute pipeline that
prevents silent or unintended agent actions. This ADR describes the
corresponding client-side UX patterns and React components that surface those
safety guarantees to the editor. The challenge is to make the human-in-the-loop
workflow feel fast and lightweight -- not like a compliance gate -- while
ensuring that no destructive action can occur without explicit user consent.

The components must work in both the Electron desktop shell and the web
application, so they are built as pure presentation components with no direct
store or service dependencies.

## Decision

### 1. Plan Preview Before Execution -- No Silent Actions

Every plan generated by the orchestrator is displayed to the user in a
`PlanPreview` drawer before any tool call executes. The drawer shows:

- The user's original intent (what they asked for).
- An ordered list of steps the agent intends to take.
- Each step's tool name and human-readable description.
- A coloured status dot for each step (pending, executing, completed, failed).
- The estimated token cost of the full plan.

The user cannot bypass the preview phase. There is no "auto-execute" mode in
the UI, even when the server-side policy is set to `auto-approve`, because the
preview still renders -- it just transitions to `approved` automatically.

### 2. Explicit Approve / Reject for Every Plan

The `PlanPreview` component presents two prominent action buttons:

- **Approve All** -- transitions the plan from `preview` to `approved` and
  begins execution.
- **Reject** -- transitions the plan to `cancelled` with no side effects.

Both actions require an explicit click or keyboard activation (Enter/Space).
There are no auto-dismiss timers, invisible form submissions, or implicit
consent mechanisms.

### 3. Per-Step Approval for Fine-Grained Control

For editors who want tighter control, `PlanPreview` supports an optional
`onApproveStep` callback that renders a per-step approve button. This allows
the user to:

- Approve and execute steps one at a time.
- Skip or reorder steps (future enhancement).
- Inspect intermediate results between steps.

Per-step approval is especially important for destructive operations like
`overwrite`, `extract`, or `export_sequence`, which are flagged in the
`ApprovalPolicy.requireApprovalFor` list.

### 4. Token Cost Transparency Before Premium Jobs

The `TokenBadge` component is displayed before any operation that consumes
tokens. It shows:

- The estimated token cost for the pending operation.
- The user's current token balance.
- A warning state when the balance is insufficient.
- An optional category label (e.g. "Assembly", "Transcribe").

The badge can be clicked to open a purchase or top-up dialog. This ensures
editors are never surprised by cost, and the system never silently deducts
tokens without prior notice.

### 5. Execution History with Undo Capabilities

The `ExecutionHistory` component provides a scrollable log of all past agent
executions. Each entry displays:

- Status icon (completed, failed, cancelled, in-progress).
- The original intent text.
- Step progress (e.g. "3/5 steps").
- Token usage.
- Relative timestamp.
- An **Undo** button for plans whose steps registered compensation functions.
- A **View** button to re-open the plan details.

This makes the agent's past actions fully transparent and reversible, reducing
the risk of accumulated silent damage.

### 6. Context Pill Shows What the Agent Is Working On

The `ContextPill` component is a compact badge that communicates the current
editing context to the user. It appears next to the prompt bar and inside the
plan preview to answer the question: "What is the agent looking at right now?"

Context types include: project, bin, selection, sequence, and clip. Each type
has a distinct colour and icon for rapid visual identification. The pill
supports click-to-inspect and remove-to-clear affordances.

### 7. Pure Presentation Components for Testability

All seven UI components (`PromptBar`, `ContextPill`, `PlanPreview`,
`ResultsPanel`, `PlaybookBuilder`, `TokenBadge`, `ExecutionHistory`) follow a
strict presentational pattern:

- Props-only interface; no direct dependency on Zustand stores, services, or
  singletons.
- All state mutations flow through callback props (`onSubmit`, `onApprove`,
  `onReject`, `onUndo`, etc.).
- Components can be rendered in Storybook, unit tests, or any React host
  without initialising the full application.
- CSS is externalised to a single `agentic-ui.css` file that consumes design
  system custom properties from the host app, with local fallbacks for
  standalone use.

This separation makes it straightforward to:

- Write deterministic tests that assert on rendered output.
- Reuse components across the web app and Electron desktop shell.
- Replace the styling layer without touching component logic.

### 8. Accessibility

All components include:

- ARIA roles and labels (`role="region"`, `aria-label`, `aria-expanded`, etc.).
- Keyboard navigation (Enter/Space to activate buttons, Tab to navigate).
- Screen-reader-friendly status announcements (`role="status"` for live
  regions).
- Focus management for interactive elements.

## Consequences

**Positive:**

- Every agent action is visible and requires explicit human approval.
- Token costs are transparent before execution.
- Past actions are logged and reversible.
- Context is always communicated, preventing "what is it doing?" confusion.
- Components are portable, testable, and host-agnostic.
- Accessibility is built in from the start.

**Negative / Trade-offs:**

- The preview-and-approve flow adds one interaction step before execution.
  This is a deliberate safety trade-off. For experienced users, the
  `auto-approve` server policy reduces this to a visual confirmation.
- Pure presentation components require the integrating host to wire up store
  connections and callbacks. This is additional glue code but keeps the
  component library decoupled.
- The `agentic-ui.css` file uses custom property fallbacks, which adds some
  duplication. This is acceptable for standalone operability.

## Alternatives Considered

1. **Store-connected smart components** -- Rejected because coupling to
   Zustand prevents reuse in Electron IPC panels and Storybook isolation.
2. **Inline styles instead of CSS classes** -- Rejected because class-based
   styling integrates with the existing design system, supports theming, and
   allows consumer overrides.
3. **Auto-execute with post-hoc notification** -- Rejected because it
   violates the core safety principle that no destructive action should occur
   without prior user consent.
4. **Per-tool approval dialogs (modal)** -- Rejected in favour of plan-level
   preview to reduce interruption frequency while maintaining safety.
